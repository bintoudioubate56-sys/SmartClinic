Projet

SmartClinic — Plateforme de gestion clinique pour la Guinée

Application web multi-rôles permettant aux cliniques privées guinéennes de gérer patients, rendez-vous et dossiers médicaux. Remplace les carnets papier par un système numérique sécurisé.

Stack technique
⚙
Next.js 14 (App Router)
Framework fullstack. Gère les pages React ET la logique serveur via Server Actions. Pas de backend Express séparé.
🗄
Supabase
PostgreSQL + Auth JWT + Storage fichiers + Edge Functions. Le "backend managé" — pas de serveur à configurer.
🎨
Tailwind CSS
Styles via classes directement dans le JSX. Aucun fichier CSS séparé créé.
📱
Dexie.js (IndexedDB)
Base de données locale dans le navigateur. Permet le mode hors-ligne quand internet coupe.
💬
AfricasTalking
API SMS pour notifications rendez-vous aux patients (+224XXXXXXXXX).

Flux d'une requête — du clic au retour
Clic utilisateur → Composant React → Server Action (actions/) → Supabase client serveur → PostgreSQL + RLS → ActionResult<T> → Re-render UI
Le middleware intercepte chaque requête AVANT qu'elle atteigne la page. Si pas de session JWT valide → redirect /login. Si mauvais rôle → redirect vers l'espace autorisé.
middleware.ts — logique de sécurité

C'est la première ligne de défense. Il s'exécute sur le serveur Edge à chaque requête vers /dashboard/*.

// middleware.ts (simplifié) export async function middleware(req: NextRequest) { // 1. Récupère la session JWT Supabase const { data: { session } } = await supabase.auth.getSession() // 2. Pas de session → login if (!session && path.startsWith('/dashboard')) { return NextResponse.redirect('/login') } // 3. Mauvais espace pour ce rôle → redirect const role = session?.user?.user_metadata?.role const allowed = roleRoutes[role] if (allowed && !path.startsWith(allowed)) { return NextResponse.redirect(allowed) } }

Row Level Security (RLS) — comment ça marche

La RLS est appliquée automatiquement par PostgreSQL à chaque requête, selon l'utilisateur connecté. Même si quelqu'un accède directement à l'URL, la DB refuse les données non autorisées.

-- Exemple : un médecin ne voit que les patients de SA clinique CREATE POLICY "doctors_see_own_clinic_patients" ON patients FOR SELECT TO authenticated USING ( auth.uid() IN ( SELECT user_id FROM clinic_staff WHERE clinic_id = patients.clinic_id AND role = 'doctor' ) );
Toujours tester la RLS avec un compte d'un autre rôle — une requête qui retourne null au lieu de données = RLS qui bloque correctement.
Triggers automatiques
⚡
generate_patient_number
Avant INSERT sur patients → génère SC-2025-00001 via sequence PostgreSQL. Tu n'as pas à le gérer dans le code.
⚡
generate_qr_token
Avant INSERT → génère un token hex 64 chars via gen_random_bytes(32). Unique et imprévisible.
⚡
updated_at trigger
Sur clinics, patients, appointments → met à jour updated_at automatiquement à chaque UPDATE.

Patterns obligatoires

Code à connaître par cœur

ActionResult<T> — pattern de retour universel

Toutes les Server Actions retournent ce type. Jamais de throw nu côté client.

// types — à définir une fois, utiliser partout type ActionResult<T> = | { success: true; data: T } | { success: false; error: string } // Dans une Server Action export async function createPatient( formData: FormData ): Promise<ActionResult<Patient>> { try { // ... logique métier return { success: true, data: patient } } catch (err) { console.error('[createPatient]', err) return { success: false, error: 'Erreur lors de la création du patient' } } } // Dans le composant React const result = await createPatient(formData) if (!result.success) { setError(result.error) // Afficher en français à l'utilisateur return } // result.data est typé ici, pas de any
Server Action sécurisée — vérification de rôle

Chaque action vérifie l'auth ET le rôle. La RLS est une 2e couche, pas la première.

"use server" // ← OBLIGATOIRE en première ligne export async function createRecord(data: RecordInput) { const supabase = createServerActionClient({ cookies }) // 1. Vérifie la session const { data: { session } } = await supabase.auth.getSession() if (!session) return { success: false, error: 'Non autorisé' } // 2. Vérifie le rôle const role = session.user.user_metadata.role if (role !== 'doctor') return { success: false, error: 'Permission refusée' } // 3. Logique métier + RLS fait le reste const { data: record, error } = await supabase .from('medical_records') .insert(data) .select() .single() // 4. Log audit await logAction(supabase, { action: 'record_created', ... }) return { success: true, data: record } }
Types TypeScript — pas de any, jamais
// ❌ INTERDIT const data: any = await supabase.from('patients').select() // ✅ OBLIGATOIRE import type { Database } from '@/types/database.types' type Patient = Database['public']['Tables']['patients']['Row'] const { data } = await supabase .from('patients') .select('*') .returns<Patient[]>()
logAction() — audit de chaque action
// lib/audit.ts export async function logAction(supabase, { userId, clinicId, action, targetTable, targetId, metadata, request }) { const ip = request?.headers.get('x-forwarded-for') await supabase.from('audit_logs').insert({ user_id: userId, clinic_id: clinicId, action, // 'patient_created' | 'login' | etc. target_table: targetTable, target_id: targetId, ip_address: ip, metadata // JSON libre : champs modifiés, etc. }) }

Où est stocké le rôle ?

Le rôle est stocké dans user_metadata.role de Supabase Auth (JWT). C'est ce que le middleware lit. La table clinic_staff stocke le rôle aussi pour la RLS PostgreSQL.

Un rôle dans le JWT n'est pas chiffré — ne jamais y stocker de données sensibles. C'est uniquement pour le routage.
